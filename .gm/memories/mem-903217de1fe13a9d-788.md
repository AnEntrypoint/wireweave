gm-method: when adding a caller-scoped CSS class to override a design-system component's own base rule (equal selector specificity, e.g. '.scope .component' vs '.scope .caller-class'), source order alone is not reliable if the SDK injects its stylesheet at runtime (e.g. via an installStyles()-style call) AFTER the caller's static <style> block -- the runtime-injected rule wins the cascade regardless of which file was authored first. Fix: qualify the caller's selector with BOTH classes (e.g. '.scope .component.caller-class') so specificity, not source order, decides the winner. Diagnose via browser witness comparing getComputedStyle on the live element against the expected rule, not by reading source alone -- a rule can be present verbatim in the page and still lose the cascade.
